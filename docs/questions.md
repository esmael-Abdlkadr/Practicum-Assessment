# questions.md — Ambiguities & Design Decisions

## 1. Scope Model for RBAC

- **Question:** The prompt specifies data-scope constraints at multiple granularities (self, own department, sub-departments, global, school/major/class/cohort). Should these scopes be encoded as a single hierarchical enum, stored as free-form strings per permission row, or modelled as separate scope columns?
- **My Understanding:** A flat string-per-permission approach keeps migrations simple for an offline single-machine deployment, while still being queryable. A normalised relational scope table would be over-engineered for this scale.
- **Solution:** `UserPermission.permission` stores a dot-separated permission string (e.g. `report:export`) and `TemporaryDelegation.scope` stores a freeform scope tag (e.g. `cohort:42`). `rbac_service.py` interprets these at runtime, and `decorators.py` provides `@require_scope` and `@permission_required` for route-level enforcement. `dept_admin` always bypasses permission checks implicitly. Trade-off: no DB-level referential integrity on scope values, but significantly simpler schema and zero schema migrations during the project lifetime.

## 2. CAPTCHA Implementation

- **Question:** The prompt requires CAPTCHA after 3 failed login attempts but does not specify the CAPTCHA type. Using a third-party CAPTCHA service (reCAPTCHA, hCaptcha) violates the "no third-party dependencies / fully offline" constraint.
- **My Understanding:** An offline math or image CAPTCHA served from the same Flask process satisfies both the security intent and the offline constraint.
- **Solution:** `auth_service.generate_captcha()` produces a simple arithmetic challenge stored in the server-side session. The login template renders it after `failed_attempts >= 3`. No external calls are made.

## 3. MFA — Online vs Offline

- **Question:** The prompt requires "optional offline MFA via TOTP codes generated by an authenticator app without any network calls." Standard TOTP libraries validate codes locally, but QR-code provisioning images are sometimes generated via external chart APIs.
- **My Understanding:** QR codes must be generated locally to honour the offline constraint.
- **Solution:** `qrcode` (PyPI) renders the QR image in-process and serves it as a `<img src="data:image/png;base64,…">` inline data URI. `pyotp` validates TOTP codes. No outbound requests.

## 4. Immutability of Audit Logs

- **Question:** The prompt says audit logs must be immutable. SQLite does not natively enforce row-level immutability.
- **My Understanding:** SQLAlchemy event hooks can intercept UPDATE/DELETE at the ORM layer and raise before any SQL is executed.
- **Solution:** `AuditLog` overrides `update()` and `delete()` and uses `@event.listens_for(AuditLog, "before_update")` / `"before_delete"` to raise `PermissionError`. Direct SQL bypasses this, which is acceptable for the threat model (single trusted process on a local machine).

## 5. Session Storage Backend

- **Question:** Flask's default cookie-based sessions expose session data client-side. Server-side session storage is needed for secure 30-minute inactivity expiry.
- **My Understanding:** `flask-session` with filesystem storage is the simplest server-side option that requires no extra services (Redis, Memcached) and works offline.
- **Solution:** `SESSION_TYPE = "filesystem"` is set in `Config`. Session files are stored in `./flask_session/` and mounted as a Docker volume. `PERMANENT_SESSION_LIFETIME = timedelta(minutes=30)` enforces inactivity expiry.

## 6. Assignment vs Quiz Workflow Separation

- **Question:** The prompt describes both an "assignment" workflow and a "quiz" workflow. It is ambiguous whether these share models or are fully separate.
- **My Understanding:** Assignments are open-ended written submissions with a single deadline; quizzes are timed, have randomised question draws, and support autosave. They have different lifecycle states and grading paths.
- **Solution:** Assignments use dedicated models (`Assignment`, `AssignmentSubmission`, `AssignmentGrade`) and routes (`/assignments/*`). Quizzes use `Paper`, `Attempt`, and `Answer` models with routes under `/quiz/*`. Grading shares a common `grading_service` but exposes separate UI routes for each type.

## 7. Student ID Masking vs Encryption

- **Question:** The prompt requires sensitive fields like student ID to be "masked in most views and encrypted at rest." This implies two separate controls.
- **My Understanding:** Encryption at rest (Fernet symmetric encryption) protects the stored value. Masking at the display layer (e.g. `****1234`) limits accidental exposure in the UI.
- **Solution:** `student_id_enc` stores the Fernet-encrypted value. A `reveal_student_id` endpoint (GET `/admin/users/<id>/reveal-student-id`) requires elevated session (dept_admin) and returns the plaintext. All other views display a masked representation via a template filter.

## 8. Delegation Expiry Default and Maximum

- **Question:** The prompt states "temporary delegations that expire automatically (default 7 days, max 30)." It does not specify what happens if a caller omits the expiry or provides a value exceeding 30.
- **My Understanding:** Silent clamping is safer than rejection for the default case; explicit rejection for values > 30 days is clearer.
- **Solution:** `create_delegation` route defaults `expires_in_days` to 7 when absent. Any value > 30 returns HTTP 400 with a descriptive error message. A background call to `rbac_service.expire_delegations()` runs at startup and on the `before_request` hook (throttled to once per hour) to deactivate expired records.

---

## 9. Anomaly Detection Scope

- **Question:** The prompt mentions "anomalous login patterns flagged on an admin dashboard." It does not define what constitutes anomalous.
- **My Understanding:** A pragmatic minimum: rapid repeated failures from the same IP or user within a short window, and login from a new device fingerprint after a successful session from a different fingerprint.
- **Solution:** `audit_service` checks consecutive failure counts and device fingerprint changes and inserts `AnomalyFlag` records. The admin dashboard at `/admin/anomalies` lists flagged events with a "mark reviewed" action.

---

## 10. Docker Single-Service vs Multi-Service Compose

- **Question:** Since the app uses SQLite (file-based), no database container is needed. Should `docker-compose.yml` still use multi-service for separation of concerns?
- **My Understanding:** A single `web` service with mounted volumes for the SQLite data file and session directory is sufficient, simplifies operations, and removes inter-container networking from the threat surface.
- **Solution:** `docker-compose.yml` declares a single `web` service. SQLite database lives in `./data/` mounted volume. A healthcheck on `GET /health` ensures readiness before traffic is served.
